    Task 3.1: Configuration Template Engine - COMPLETION REPORT¶

    Status: ‚úÖ COMPLETE

    Executive Summary¶

    Successfully implemented a robust configuration template engine for CORTEX that enables dynamic variable substitution in JSON, YAML, and Markdown templates. The engine supports both required and optional variables with defaults, auto-detects project context, and provides comprehensive error handling.

    Deliverables¶

    1. config-template-engine.ps1 (~300 lines)¶

    Complete PowerShell module with three main functions:

    Core Function: Expand-ConfigTemplate¶

    Purpose: Expand templates with variable substitution

    Parameters: - -TemplatePath - Path to template file (parameter set 1) - -TemplateContent - String content (parameter set 2) - -Variables - Hashtable of custom variables - -OutputPath - Optional file output path - -Force - Overwrite existing files

    Features: - ‚úÖ Dual input modes (file or string) - ‚úÖ Regex-based variable substitution: {{VARIABLE}} - ‚úÖ Default value support: {{VARIABLE:default}} - ‚úÖ Required variable validation (error if missing) - ‚úÖ File output with directory creation - ‚úÖ Interactive overwrite confirmation

    Helper Function: Get-BuiltInVariables¶

    Purpose: Auto-detect project context variables

    Built-in Variables: - PROJECT_ROOT - Git workspace root (from workspace-resolver.ps1) - PROJECT_NAME - Project folder name - KDS_ROOT - CORTEX installation directory - GIT_BRANCH - Current git branch - TIMESTAMP - ISO 8601 timestamp (auto-generated)

    Features: - ‚úÖ Graceful fallback if git unavailable - ‚úÖ Integration with workspace-resolver.ps1 - ‚úÖ Custom variables override built-ins

    Validation Function: Test-TemplateVariables¶

    Purpose: Pre-flight validation of template requirements

    Features: - ‚úÖ Scans template for all variables - ‚úÖ Identifies missing required variables - ‚úÖ Returns true/false with warnings - ‚úÖ Helps prevent runtime errors

    2. ConfigTemplateEngine.tests.ps1 (~450 lines)¶

    Comprehensive Pester test suite with 40+ test cases:

    Test Coverage:¶

    • ‚úÖ Basic substitution - Single, multiple, repeated variables
    • ‚úÖ Default values - Use defaults, override defaults, empty defaults

    Key Features Implemented¶

    Variable Syntax¶

    {{VARIABLE}}         ‚Üí Required variable (throws error if missing)
{{VARIABLE:default}} ‚Üí Optional variable (uses default if missing)

    Supported Formats¶

    • ‚úÖ JSON - Preserves structure, validates syntax
    • ‚úÖ YAML - Maintains indentation and formatting
    • ‚úÖ Markdown - Preserves code blocks and formatting
    • ‚úÖ Plain text - Any text-based format

    Variable Naming Rules¶

    • ‚úÖ Must start with uppercase letter or underscore
    • ‚úÖ Can contain uppercase letters, numbers, underscores
    • ‚úÖ Examples: PROJECT_ROOT, API_URL, VAR_123
    • ‚ùå Lowercase rejected: {{lowercase}} preserved as-is

    Smart Defaults¶

    • ‚úÖ Auto-detection of project context
    • ‚úÖ Git branch detection
    • ‚úÖ Workspace root resolution
    • ‚úÖ ISO 8601 timestamps
    • ‚úÖ Graceful fallback if detection fails

    Examples¶

    Example 1: Simple Config Generation¶

    $template = @"
{
  "name": "{{PROJECT_NAME}}",
  "version": "{{VERSION:1.0.0}}"
}
"@

Expand-ConfigTemplate -TemplateContent $template -OutputPath "config.json" -Force

    Output: config.json 

    {
  "name": "CORTEX",
  "version": "1.0.0"
}

    Example 2: Custom Variables¶

    $vars = @{
    API_URL = "https://api.example.com"
    TIMEOUT = "30"
    RETRY = "3"
}

Expand-ConfigTemplate `
    -TemplatePath "service.template.yaml" `
    -Variables $vars `
    -OutputPath "service.yaml" `
    -Force

    Example 3: Pre-flight Validation¶

    if (Test-TemplateVariables -TemplatePath "config.template.json") {
    Expand-ConfigTemplate -TemplatePath "config.template.json" -OutputPath "config.json"
} else {
    Write-Warning "Missing required variables"
}
