## Executive Summary

**Problem:** Documentation fragmentation across CORTEX repository (scattered markdown files, obsolete backups, mixed user-facing and internal docs)

**Solution:** Enhanced Hybrid Structure
- **User-facing documentation:** `/docs` (architecture, guides, narratives, governance, reference)
- **Internal strategic docs:** `/cortex-brain/documents` (reports, analysis, planning, conversation-captures)
- **Cross-linking:** EPM generates `/docs` content from `/cortex-brain` sources
- **Source mapping:** TRUTH-SOURCES.yaml tracks relationships

**Decision:** User accepted Enhanced Hybrid Structure (Option A) - architecturally sound approach that preserves MkDocs compatibility, follows industry standards, maintains semantic separation, and builds on existing correct structure.

---

## Architecture Rationale

### Why Enhanced Hybrid Structure?

**4 Major Problems with Brain-Centric Approach:**

1. **Architectural Violation:** Mixes strategic memory (brain tier storage) with product documentation (user-facing guides)
2. **Tooling Incompatibility:** MkDocs expects `/docs` by convention; moving breaks ecosystem assumptions
3. **Developer Experience:** Non-standard location confuses contributors unfamiliar with CORTEX internals
4. **Existing Reality:** Already have correct two-tree structure (28 files in `/docs`, 7 categories in `/cortex-brain/documents`)

**Enhanced Hybrid Structure Benefits:**

✅ **Preserves MkDocs Compatibility:** No `docs_dir` reconfiguration needed  
✅ **Follows Industry Standards:** GitHub Pages, Read the Docs, Sphinx all expect `/docs`  
✅ **Maintains Semantic Separation:** Product docs ≠ strategic memory  
✅ **Builds on Existing Structure:** Leverage what already works correctly  
✅ **Enables Cross-Linking:** EPM generates public docs from private brain sources

#### Automated Sync Script

**Create:** `scripts/sync_documentation_structure.py`

**Purpose:** Automatically sync documentation structure when brain sources change

**Triggers:**
- Watch `cortex-brain/brain-protection-rules.yaml` for changes → regenerate THE-RULEBOOK.md
- Watch `cortex-brain/module-definitions.yaml` for changes → regenerate EXECUTIVE-SUMMARY.md
- Watch test results for changes → update executive summary metrics

**Implementation:**

```python
import os
import time
from watchdog.observers import Observer
from watchdog.events import FileSystemEventHandler

class DocumentationSyncHandler(FileSystemEventHandler):
    def on_modified(self, event):
        if event.src_path.endswith('brain-protection-rules.yaml'):
            print("Brain protection rules changed. Regenerating THE-RULEBOOK.md...")
            RulebookGeneratorComponent().generate()
        
        elif event.src_path.endswith('module-definitions.yaml'):
            print("Module definitions changed. Regenerating EXECUTIVE-SUMMARY.md...")
            ExecutiveSummaryComponent().generate()

if __name__ == "__main__":
    event_handler = DocumentationSyncHandler()
    observer = Observer()
    observer.schedule(event_handler, path='cortex-brain', recursive=True)
    observer.start()
    
    print("Documentation sync watcher started...")
    try:
        while True:
            time.sleep(1)
    except KeyboardInterrupt:
        observer.stop()
    observer.join()
```

## Risk Mitigation

### Risk 1: Broken Internal References

**Mitigation:**
- Run link validation before committing changes
- Use relative paths in markdown (not absolute)
- Test MkDocs site build after each phase
- DocumentValidator pre-flight checks

### Risk 2: Missing Narrative Content

**Mitigation:**
- Check if narrative chapters exist before moving
- Generate placeholder chapters if missing
- NarrativeGeneratorComponent creates missing content
- Manual review before Phase 3 completion

### Risk 3: EPM Component Implementation Delays

**Mitigation:**
- Start with simplest component (RulebookGenerator)
- Test each component independently before integration
- Use existing component patterns (DiagramGenerator) as templates
- Parallel development of 3 components possible

### Risk 4: CI/CD Integration Issues

**Mitigation:**
- Test health checks locally before CI/CD integration
- Add health check to manual validation first
- Gradual rollout (warning mode before blocking mode)
- Fallback to manual validation if CI/CD fails
