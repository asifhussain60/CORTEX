    Database Crawler - SQL File Priority Enhancement¶

    Date: 2025-11-04
    Version: 1.1.0
    Status: ‚úÖ Implemented

    üéØ Enhancement Summary¶

    The database crawler now prioritizes discovering SQL schema and data files BEFORE attempting database connections. This makes it easier and faster for the brain to understand database structure without needing live database access.

    üìã Discovery Priority (New Workflow)¶

    Priority 1: SQL Schema/Data Files (Fastest, Easiest) ‚ö°¶

    Scans for: - *schema*.sql - Schema/DDL files - *ddl*.sql - Data Definition Language - *create*.sql - CREATE TABLE statements - *structure*.sql - Database structure - *data*.sql - Data/DML files - *dml*.sql - Data Manipulation Language - *insert*.sql - INSERT statements - *seed*.sql - Seed data

    What it extracts: - ‚úÖ File type (schema/data/mixed) - ‚úÖ Statement counts (CREATE TABLE, INSERT, etc.) - ‚úÖ Referenced table names - ‚úÖ File size and line counts - ‚úÖ DDL vs DML distribution

    Benefits: - ‚úÖ No database connection needed - Works offline - ‚úÖ Fast - File scanning vs database queries - ‚úÖ Safe - No credentials, no network access - ‚úÖ Version controlled - SQL files in Git = history - ‚úÖ Brain-friendly - Easy to reference in prompts

    Priority 2: Configuration Discovery¶

    Scans for: - Connection strings (appsettings.json, web.config, .env) - Database provider detection - Entity models and DbContext files - Migration files

    Priority 3: Live Database Connection (Only if Needed)¶

    Triggers connection only if: - ‚ùå No SQL schema files found AND - ‚úÖ Connection string available

    OR

    • ‚úÖ Explicitly requested via -ConnectAndCrawl flag

    If no connection string: - Prompts user to provide one - Memorizes it to KDS/kds-brain/database-connection.txt - Reuses for future crawls

    ‚úÖ Benefits Summary¶

    For Users¶

    • ‚úÖ Faster - 30 seconds vs 2-5 minutes for database connection
    • ‚úÖ Easier - No connection string needed
    • ‚úÖ Safer - No credentials exposed
    • ‚úÖ Offline - Works without database access

    For Brain¶

    • ‚úÖ Better references - File paths instead of "live database"
    • ‚úÖ Version history - SQL files in Git = timeline
    • ‚úÖ Complete context - Sees CREATE statements, not just column lists
    • ‚úÖ Relationship aware - Sees FOREIGN KEY definitions

    For Projects¶

    • ‚úÖ Portable - SQL files travel with code
    • ‚úÖ Documented - Schema is self-documenting
    • ‚úÖ Reproducible - Anyone can run SQL files
    • ‚úÖ Reviewable - Schema changes in pull requests

    üìù Recommended Practices¶

    1. Create SQL Schema Files¶

    Even if your project uses Entity Framework migrations, create readable SQL files:

    /database
  /schema
    - schema.sql          (Main schema - CREATE TABLE statements)
    - indexes.sql         (All indexes)
    - constraints.sql     (Foreign keys, constraints)
    - views.sql           (Views and stored procedures)
  /data
    - seed-data.sql       (Development seed data)
    - test-data.sql       (Test fixtures)

    2. Keep SQL Files Updated¶

    When schema changes: - ‚úÖ Update the SQL file - ‚úÖ Commit with migration - ‚úÖ Re-run crawler: .\database-crawler.ps1 -WorkspaceRoot .

    3. Name Files Clearly¶

    Good names: - ‚úÖ database-schema.sql - ‚úÖ 01-schema-initial.sql - ‚úÖ seed-data-users.sql

    Bad names: - ‚ùå temp.sql - ‚ùå new.sql - ‚ùå backup-copy-2.sql

    4. Version Control¶

    Add to .gitignore (if needed): 

    # Exclude personal local database configs
appsettings.local.json
database-connection.txt

# Include schema files
!database/schema/*.sql
!database/data/*.sql

    üîÑ Migration Guide¶

    If You Have EF Migrations Only¶

    Extract schema to SQL:

    # Generate SQL from migrations
dotnet ef migrations script --output database/schema.sql

# Clean up the file (remove migration metadata if needed)

    If You Have Database But No SQL Files¶

    Export schema:

    -- SQL Server: Generate CREATE scripts via SSMS
Right-click Database ‚Üí Tasks ‚Üí Generate Scripts ‚Üí Select tables

-- Or use sqlcmd
sqlcmd -S localhost -d MyDB -Q "SELECT * FROM INFORMATION_SCHEMA.TABLES" -o schema-export.sql

    Save as database/schema.sql and commit.

    üéì Technical Details¶

    SQL File Parsing Logic¶

    Type Detection: 

    IF (DDL > 0 AND DML == 0) ‚Üí type = "schema"
IF (DML > 0 AND DDL == 0) ‚Üí type = "data"
IF (DDL > 0 AND DML > 0)  ‚Üí type = "mixed"

    Table Extraction: - Regex: \b(?:FROM|INTO|TABLE|UPDATE|JOIN)\s+(?:\[?(\w+)\]?\.)?(?:\[?(\w+)\]?) - Filters SQL keywords (SELECT, WHERE, etc.) - Captures schema.table or just table

    Statement Counting: - Case-insensitive regex matches - CREATE TABLE, ALTER TABLE, DROP TABLE - INSERT INTO, UPDATE SET, DELETE FROM - CREATE INDEX, CREATE VIEW, CREATE PROCEDURE

    üìä Performance¶

    Typical Performance: - SQL file scan: 5-30 seconds (depending on file count) - Config discovery: 10-30 seconds - Live DB connection: 1-5 minutes (if needed)

    Total: - With SQL files: 15-60 seconds ‚ö° - Without SQL files: 2-6 minutes

    Speedup: ~10x faster when SQL files exist!
